Purpose of Command

The SECMIGR command assists you in converting your current security settings and definitions. The command produces output members which can be used to define the desired security settings. In most cases SECMIGR converts existing security statements into statements that provide equivalent security. Review the generated statements to validate that they provide the desired security before enabling them in your installation.

The SECMIGR command will perform system symbolic substitution on records read from disk. The NetView SUBSYM PIPE stage is added to all PIPE commands that have disk read stages.

Substitution is always performed on the &DOMAIN symbolic, unless substitution was disabled when the NetView program was started. For MVS and user-defined system symbolics, substitution is not performed if substitution was disabled when the NetView program was started, or if you have not defined an MVS system symbolic on your system.

Entering the SECMIGR command with no operands initiates a full-screen interface to prompt you for your processing options. To bypass this set of panels, enter the options on the command line.

Specifies that a double asterisk and a period (**.) are prefixed to each resource name as it is added tothe list of resources for a span. This list of resources is then used to create the SPANDEF statement. Avalue other than Y prevents a double asterisk from being added. The default value is Y.   • When SECMIGR creates output members, it places a 2-character key field (>S) in the first line of the report. If you attempt to replace a member or file that does not have the key field in the correct location, you will receive an error message. You can prevent a SECMIGR output member from being overwritten by removing the write protect key from the first line of the member.
  • The output member of the SECMIGR command using either the SPN2TBL or the SCP2TBL function must be placed in a data set in the DSIPARM concatenation before it can be used by the NetView program.
  • When the SECMIGR command creates an output member, it includes comments that include the operator ID that issued the command, the date and time it was run, the input DD name and member names, and the output data set and member name.
  • For SECMIGR requests using the OPS2RACF, SCP2RACF, SPN2RACF, and TBL2RACF functions, the output member contains RACF commands. The output member formatting must be altered in order to place the statements into effect using a batch job or TSO CLIST.

    SEQUENT
     Syntax SEQUENT

    Purpose of Command

    The SEQUENT command allows for the serialization of operations in a programmatic environment, similar to the ENQ and DEQ assembler macros in MVS. A named resource is specified to obtain, release, monitor, or list. Obtaining a resource as exclusive prevents other SEQUENT obtain requests for the same resource name to be satisfied, until the resource is released by the program that obtained it. Obtain requests that cannot be immediately satisfied cause the running program to be suspended. At the instant when the resource is released, the next waiting obtain request is satisfied, and the requesting program resumes. If a resource is requested to be obtained as shared, other shared requests can also be satisfied, as long as an exclusive request is not queued before the shared request.

    Obtain requests are satisfied in the order they are requested, unless the program environment gets in a state where it cannot take ownership of the resource, in which case the next waiting request is offered ownership of the resource. An example of this would be when a high priority command interrupts the program requesting the SEQUENT resource.

    The SEQUENT command can also be used to perform the following tasks:

  • Test the current state of a SEQUENT resource name in a programmatic environment. The state of the resource is indicated by a return code.
  • Monitor actions taken upon one or more resource names, which include obtaining, releasing, and waiting for ownership of a resource. Each of these actions is indicated by a message.
  • List one or more resource names to show the current status of ownership or request, and how long it has been in that state. This list can be filtered by amount of time held (obtained), waiting, or by requesting operator ID.

    • Operand Descriptions HELD

    Limits the display of SEQUENT names that are listed to those names that are currently held and owned by some task, and for at least the amount of time specified. To list all that are held, specify HELD with a value of 0 or no value at all.

    LIST

    Causes information about the specified SEQUENT names to be displayed, which could be more than one if wild cards are specified. Messages that contain the following information are presented to the operator who issues the command:

  • SEQUENT name
  • Task name that is currently holding the SEQUENT resource or blank if currently not being held
  • Task names that are currently waiting for the SEQUENT resource (on separate lines)
  • Indication as to whether the SEQUENT resource is being held or being waited upon or available
  • Date or time when the SEQUENT resource was obtained or started waiting for the SEQUENT resource or became available
  • The length of time that the SEQUENT resource has been obtained, waiting, or been available This option is not restricted to procedure environments.

    • MONITOR

    Specifies a list of SEQUENT resource filters, each of which specifies that a set of SEQUENT names is or is not to be monitored for activity. The following actions will cause a message to be issued, which is also automatable, for SEQUENT names that are being monitored:

  • Successfully obtaining a SEQUENT resource
  • Being queued up on an OBTAINEX or OBTAINSH request
  • Successfully releasing a SEQUENT resource

    • Multiple filters, with or without wild cards, and with or without a 'not' sign ('¬') as the first character, can be specified on a single command. Successive SEQUENT commands with MONITOR result in a cumulative effect of the filters, which are scanned in the order specified to determine whether a SEQUENT name matches. The last filter that a SEQUENT name matches determines whether a monitoring message is issued for it. To remove all monitoring, specify MONITOR=¬*. A filter specified following the ¬*, even on the same invocation of MONITOR, behaves as if there were no filters in place before it. This option is not restricted to procedure environments.

    MONLIST

    Causes the filters in the current MONITOR list, as well as the current MONOPER value, to be displayed. This option is not restricted to procedure environments.

    MONOPER

    Specifies an operator ID to which monitoring messages are sent, in addition to being logged to NETLOG and Canzlog. If * is specified, the current operator ID is used. If *NONE* is specified, monitoring messages are only logged to NETLOG and Canzlog. If MONOPER is not specified together with MONITOR, the monitoring operator ID remains the same as it had been set from a previous specification of MONOPER. If no previous MONOPER had been specified, and MONOPER is not specified, the operator that issues the SEQUENT MONITOR command is sent the monitoring messages.

    OBTAINEX

    Requests exclusive ownership of the resource specified on seqname. If the current command group currently owns the resource as shared, ownership is converted to exclusive, if no other task request currently owns it as shared; otherwise, the request fails with a specific return code. Control is not returned to the issuing code until the resource is available or an error is detected. Ownership is assigned based on the order of each request, as long as the command group is available to take ownership. This option may only be used when running in a procedure environment. Unless the option is not issued from within a procedure environment, in which case DSI290I is issued and the command

    fails, or a syntax or environmental error is detected, no message is issued, and the only error indication is in the form of a return code. See the list ofreturn codes for details.

    OBTAINSH

    Requests shared ownership of the resource specified on seqname. Control is not returned to the issuing code until the resource is available or an error is detected. For this request, other command groups in the same or other tasks may also own the resource as shared. Ownership is assigned based on the order of each request, as long as the command group is available to take ownership. This option may only be used when running in a procedure environment. Unless the option is not issued from within a procedure environment, in which case DSI290I is issued and the command fails, or a syntax or environmental error is detected, no message is issued, and the only error indication is in the form of a return code. See the list of return codes for details.

    OPID

    Limits the display of SEQUENT names that are listed to those that are either held by or being waited upon by the specified operator ID.

    opid

    A 1 - 8 character operator ID or function name that is preceded by a question mark (?) that resolves to an operator ID. The ID may not contain wild card characters. When the operator ID is specified on the OPID keyword with the LIST action, the specified operator ID limits the output to SEQUENT resources that are holding or waiting for a matching name. When the operator ID is specified on the MONOPER keyword with the MONITOR action, monitoring messages are sent to the specified operator ID, or are only logged if *NONE* is specified for opid.

    RELEASE Releases the resource specified on seqname, if it is currently owned by the current command group. This option may only be used when running in a procedure environment. Unless the option is not issued from within a procedure environment, in which case DSI290I is issued and the command fails, or a syntax or environmental error is detected, no message is issued, and the only error indication is in the form of a return code. See the list ofreturn codes for details. If a SEQUENT resource had not been released by a SEQUENT RELEASE command when the command group ends or the operator logs off, the resource is automatically released, and a warning message is issued. seconds

    The number of seconds to limit the LIST output to. Only those SEQUENT resources that are held or waiting at least the number of seconds specified will be listed. The format of seconds can be one of the following formats:

  • ss (number of seconds, up to 3599999)
  • mm:ss (number of minutes up to 59999 and seconds up to 59)
  • hh:mm:ss (number of hours up to 999, minutes up to 59 and seconds up to 59)

    • seqname

    The 1 - 16 character name of a SEQUENT resource to obtain, release, or test. Only the following characters are allowed in the name:

    A-Z a-z 0-9 . # @ ¢ $ _ ! ?

    The name is case-sensitive, which means that a resource name of ABC is a different resource from one named abc.

    seqnamew

    A SEQUENT name that follows the character limitations that are documented for seqname that can also contain wild cards: asterisk (*) for multicharacter and percent sign (%) for single character.

    TEST

    Tests whether the resource specified on seqname is known to the NetView program, and if so, whether it is currently owned, and in what manner. This option may only be used when running in a procedure environment. Unless the option is not issued from within a procedure environment, in which case DSI290I is issued and the command fails, or a syntax or environmental error is detected, no message is issued, and the current status of the SEQUENT resource is returned in the form of a return code. See the list of return codes for details.

    WAIT

    Limits the display of SEQUENT names that are listed to those names that are currently being waited upon and requested to be obtained, but the OBTAINEX or OBTAINSH request hasn't been satisfied yet, and for at least the amount of time specified. To list all that are being waited upon, specify WAIT with a value of 0 or no value at all.

    Usage Notes

  • Obtain requests are satisfied in the order that they are requested, unless the requesting command procedure group is not in control when the requested resource becomes available. This case can occur, for example, when a high priority command interrupts the command group that is waiting for a resource.
  • When you specify a SEQUENT name that contains lower-case characters, ensure that the SEQUENT command is executed in a NETVASIS environment.
  • Use caution when requesting ownership (with OBTAINEX or OBTAINSH) when running under a high priority command. Doing so can result in a lockout condition if the high priority command cannot obtain the resource immediately and the command group (running under a different task) that owns the resource is dependent on an action taken by a command that is interrupted by the high priority command.
  • For a more in-depth discussion of various considerations for using the SEQUENT command, see Using the SEQUENT Command to Serialize Access to Resources in the IBM Z NetView Application Programmer's Guide.
